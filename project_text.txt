General Project Description
Software testing is an important aspect of creating reliable software. Therefore, the ﬁnal project is
the main component of this course. The objectives of this project are to prepare you (as a member
of a software development team) to detect errors, faults and failures in a given application and
plan to rectify the problems using the methods you learned in his course.
Project Composition: The project will be completed by a group of 4 to 5 students as a team.
A software or web-based open-source application should be selected by each group as the project
topic. The size of the selected project should be reasonable and large enough to simulate enterprise
software testing which will be further described in the following sections.
Moreover, the project should be performed based on a proposed test plan by each group. Each
group needs to propose a test plan containing the tools and testing methods covered by the course
to be used for testing the selected Java open-source application. The selected project topic and the
test plan should be submitted and approved by the course staﬀand/or the course instructor. The
test plan should contain the general information application (including open source URL, number
of classes and methods, number of units, and major software features), applicable testing methods,
and required testing tools for each test method.
LLM-based Software Testing: In addition to regular software testing activities, each group
will conduct a small experiment on using Large Language Models (LLMs) as testing assistants. In
particular, students will interact with an open-source code LLM (such as Code Llama, StarCoder,
or GPT) to generate or reﬁne test cases for selected Java methods in their chosen application.
The goal is to critically evaluate how LLM-generated tests compare to manually designed tests in
terms of coverage, fault detection, and overall reliability, and to analyze/reﬂect on the beneﬁts and
limitations of LLM-based software testing. An example of such a code-oriented LLM is StarCoder,
a 15B-parameter model trained on source code from 80+ programming languages.
A ﬁnal written report of the project is also required to successfully complete the project. The ﬁnal
written report should contain the ﬁnalized test plan, all ﬁndings, critical analysis, barriers, and
results achieved by applying diﬀerent testing methods and tools on the open-source application.

Resources and Testing Tools
The following testing tools and resources can help you develop your project testing strategies:
• JUnit 4: A unit testing tool.
• Selenium: A testing tool for web-based application and browser testing.
• JMock: A testing tool for test-driven development in Java to test the interactions between
the objects.
• JMeter: The Apache JMeter application is an open source software designed to load test
functional behavior and measure performance. It was originally designed for testing web
applications but has been expanded to other test functions.
• TestNG: A unit testing tool inspired from JUnit and NUnit but introducing some new func-
tionalities that make it more powerful and easier to use.
• Mockito: A unit testing tool that lets you write elegant tests with a clean and simple API.
• PowerMock: A unit testing tool with more powerful capabilities.
• µJava: muJava (µJava) is a mutation system for Java programs. It automatically generates
mutants for both traditional mutation testing and class-level mutation testing. It can test
individual classes and packages of multiple classes.
• LLM-based Testing Assistants: Open-source large language models specialized for code, such
as StarCoder or Code Llama. These models can generate unit tests, suggest input partitions,
or provide explanations about code under test. Students are expected to use at least one such
model in their project in an automated manner (i.e., not directly prompting through LLMs)
and critically evaluate its outputs using the testing techniques introduced in this course.
• Introductory material on using LLMs in software engineering and software testing, for ex-
ample:
– StarCoder models and documentation
– StarCoder blog post
– Code Llama overview
– A recent survey1 on LLM-based test generation and automated software testing titled
“A Review of Large Language Models for Automated Test Case Generation”
– A recent paper2 on LLM-based integration of mutation testing into unit test generation
– Public benchmarks and datasets for testing LLMs on test-case generation tasks, e.g.,

5.1
Testing Abstractions/Approaches
Groups should apply each one of the following testing abstractions/approaches to 10
diﬀerent methods in at least 5 diﬀerent Java classes or at a similar complexity level
in Python along with incorporating LLM-assisted testing component detailed in Section A.
For each test, you will determine test values and use Unit Testing. Based on your project topic
and application, you may use standard JUnit, JUnit Parameterized, JUnit Theories, and TestNG.
Obviously, you will implement/code your testing programs.
You will need to cover 4 diﬀerent testing abstractions out of the following 5 test methods in your
ﬁnal project:
1. Input Space Partitioning (ISP): You will need to use Boundary Value Analysis (BVA)
to test your input space based on the application of a Java class. Obviously, you will imple-
ment/code your testing programs.
2. Graph-based testing (CFG and DFG): You will need to draw CFGs and provide the
corresponding analysis based on diﬀerent coverage methods for control ﬂow. You will need
to draw DFGs and provide the corresponding analysis based on diﬀerent coverage methods
for data ﬂow.
3. Logic-based Testing: You will need to provide the logical analysis based on diﬀerent
coverage methods for logical predicates/clauses.
4. Mutation Testing: You will need to modify the program under test in small ways, e.g.,
ﬂipping a less-than sign to a greater-than or changing a hard-coded number from 0 to 1.
These are usually designed to mimic typical programming errors, such as typographical errors,
wrong choice of operator, or oﬀ-by-one errors.
5. Formal Methods: You will need to select a portion of the program (e.g., speciﬁc safety-
critical methods), provided with justiﬁcations, to conduct veriﬁcation methods and prove
the program or a speciﬁc method is behaving as expected. The selected method should be
complex enough to cover diﬀerent aspects of formal veriﬁcation.

Manual Test Design
For each selected class and method under test, students must ﬁrst design a manual test suite using
the techniques from the rest of the project. In particular, for each chosen method:
• Deﬁne test requirements using input space partitioning (ISP), including appropriate parti-
tions and boundary values.
• Derive concrete test inputs to satisfy the relevant control-ﬂow and data-ﬂow coverage criteria
(CFG/DFG) introduced in the course, where applicable.
• Apply logic-based coverage criteria to predicates and clauses (e.g., condition/decision cover-
age), when meaningful.
• Implement the resulting test cases as JUnit tests under src/test/java using JUnit 5.
This manually designed test suite will serve as the baseline for comparison with the LLM-generated
tests produced by the ChatUniTest framework.
Using an LLM as a Testing Assistant via ChatUniTest
After constructing the manual test suite, each group must use the provided ChatUniTest frame-
work, described in Section A, to generate additional tests with a code-oriented LLM (e.g.,
codeqwen:v1.5-chat conﬁgured via a local OpenAI-compatible endpoint). You do not need to
interact with the LLM directly; instead, you will drive it through Maven using ChatUniTest.
Conceptually, ChatUniTest constructs prompts that include the class under test (signatures, method
bodies, and surrounding context) and asks the LLM to propose and reﬁne JUnit test cases. You
are not required to hand-craft prompts, but you will understand that the LLM is being used to
generate candidate tests based on the code and its intended behavior.
Evaluation and Comparison
You must evaluate the LLM-generated tests using the same techniques and tools that you apply
to your manually designed tests. At a minimum, for each selected class, you should:
• Structural coverage. Measure statement and branch coverage for:
1. the manually written test suite (manual tests only);
2. the LLM-generated test suite (tests produced by ChatUniTest only);
3. the combination of both (manual + LLM-generated tests).
Use the coverage tools already employed in the rest of the project.
• Mutation-based evaluation. Run the existing mutation testing setup included with the
project (e.g., PIT or the conﬁgured mutation tool) to compute the mutation score for:
1. the manual test suite, and
2. the combined manual + LLM-generated test suite.

LLM-assisted Testing: ChatUnitest Maven Plugin
Prior work has explored using large language models (LLMs) to support a range of programming
tasks, often with promising outcomes. However, using an LLM in a direct-prompting workﬂow
introduces practical constraints. First, generated code frequently fails to run as-is, echoing the
common experience of “ﬁve minutes of coding, two hours of debugging.” Second, incorporating
model outputs into an existing codebase can be inconvenient, since it typically requires manual,
out-of-band interaction and repeated context switching across tools and pages. To mitigate these
issues, the ChatUniTest framework3 proposes a Generate-Validate-Fix workﬂow and implements
a prototype system, along with plugins designed to integrate smoothly into established development
pipelines. The Maven plugin is now published to the Maven Central Repository for your trial and
feedback. Additionally, the Chatunitest plugin has been launched in the IntelliJ IDEA Plugin
Marketplace. You can search for and install ChatUniTest in the marketplace or visit the plugin
page Chatunitest: IntelliJ IDEA Plugin for more information. You can check out the ChatUniTest
repository for more details on the properties and conﬁguration parameters.
You will run an open-source code model locally using a tool such as Ollama or LM Studio, which
exposes an OpenAI-compatible HTTP endpoint on the local machine. In this case, the ChatUniTest
plugin is conﬁgured to call the local endpoint (e.g., http://localhost:11434/v1/chat/completions)
with a dummy API key, and no external paid API is required.

Q6: Why does mvn -Dtest=*Calculator* test run Calculator_Suite with N tests, but
plain mvn test does not?
ChatUniTest typically generates:
• several test classes (e.g., Calculator_add_0_0_Test), and
• an aggregate suite class (e.g., Calculator_Suite) that collects and executes these tests.
By default, Sureﬁre only includes ﬁles matching patterns like:
**/Test*.java, **/*Test.java, **/*Tests.java, **/*TestCase.java
The suite Calculator_Suite.java does not match these patterns, so plain mvn test may
ignore it.
When running:
mvn -Dtest=*Calculator* test
Sureﬁre is instructed to run classes whose names match *Calculator*, which explicitly in-
cludes Calculator_Suite. The JUnit platform then discovers and executes the N underlying
tests referenced by the suite.
To make plain mvn test also run the suite, instructors can conﬁgure Sureﬁre to include suites
explicitly:
<includes>
<include>**/*Test.java</include>
<include>**/*Tests.java</include>
<include>**/*TestCase.java</include>
<include>**/*Suite.java</include> <!-- LLM-generated suites -->
</includes>
22
Q7: What is the warning “Ignoring test class using JUnitPlatform runner: . . . Suite”
and is it a problem?
When running tests, students may see:
WARNING: Ignoring test class using JUnitPlatform runner:
com.example.Calculator_Suite
This comes from JUnit’s Vintage engine: the suite is annotated with @RunWith(JUnitPlatform.class),
which is an older JUnit 4 style bridge to the JUnit 5 platform. When running directly on the
JUnit 5 platform (with JUnit 5 and a modern Sureﬁre plugin), this annotation is redundant,
and the engine logs that it is ignoring it.
This warning does not indicate a failure. As long as Maven reports a non-zero number of
tests run and BUILD SUCCESS, the suite and all underlying tests have executed correctly. The
warning can safely be ignored for the purposes of this project.
Q8: Do I always need a suite (e.g., Calculator_Suite) to get non-zero tests?
In general, JUnit tests do not require a suite; a simple class such as FooTest with @Test
methods is suﬃcient.
However, in the current ChatUniTest setup used in this project, the generated suite (e.g.,
Calculator_Suite) often acts as the canonical entry point for the LLM-generated tests. The
suite collects multiple generated test classes and presents them as a single test container to
the JUnit platform. If the suite is not discovered (by default includes or an explicit pattern)
and the individual *Test classes are not treated as standalone containers by the current
conﬁguration, mvn test may report zero tests.
For simplicity, this project assumes that:
• the generated *_Suite class is the main entry point for LLM-generated tests, and
• Sureﬁre includes both *Test.java and *Suite.java patterns.
Q9: Why are tests like Calculator_add_0_0_Test.java not detected directly when I
ignore the suite?
The Sureﬁre <include> patterns (e.g., **/*Test.java) only control which classes are con-
sidered as candidates for testing; they do not guarantee that these classes will contribute
tests. After the ﬁle-pattern ﬁlter, the JUnit engines (Jupiter, Vintage, etc.) perform test
discovery inside each candidate class.
In this setup, the generated Calculator_add_0_0_Test classes are designed to be executed
via the suite (e.g., Calculator_Suite, which references them using @SelectClasses). When
the suite is run, the JUnit platform discovers and executes the underlying tests. When the
suite is ignored, the engines may not treat those helper classes as standalone test containers
under the current conﬁguration, resulting in zero tests. Therefore, the recommended practice
in this project is to run the generated suite class (via *Suite.java includes) rather than
attempting to run each *_Test ﬁle directly.
Q10: Can I add a speciﬁc version of JUnit to “ﬁx” the suite warning? Can I use
both JUnit 4 and JUnit 5 in the same project?
The warning about ignoring the JUnitPlatform runner is not caused by an incompatible
JUnit version. It is produced because the suite uses an older JUnit 4 bridge annotation
(@RunWith(JUnitPlatform.class)) while the tests are already executed on the JUnit 5
platform.
Adding a speciﬁc JUnit 4 dependency (e.g., junit:4.13.2) will not remove this warning. In
fact, the presence of JUnit 4/Vintage support is what enables this message.
Recommended conﬁguration for this project:
• Use JUnit 5 as the primary test framework:
<dependency>
<groupId>org.junit.jupiter</groupId>
23
<artifactId>junit-jupiter</artifactId>
<version>5.9.2</version>
<scope>test</scope>
</dependency>
• Use a modern Sureﬁre plugin (e.g., version 3.1.2) conﬁgured for JUnit 5.
It is technically possible to include both JUnit 4 and JUnit 5 (using the Vintage engine to
run JUnit 4 tests), but this is not required for the project and does not eliminate the suite
warning. For the purposes of this course, students may safely ignore this warning and rely
on JUnit 5 as described above.
Q11: What if PIT (pitest-maven) fails with environment errors?
On some machines, the PIT plugin may fail with environment-speciﬁc errors (e.g., about the
“coverage generation minion” exiting with UNKNOWN_ERROR). If the provided lab or starter
environment runs PIT successfully, students should use PIT as described in the mutation
testing section of the manual.
If PIT cannot be run on a particular machine, students must still perform a manual mutant
experiment:
1. Start from the correct implementation and run mvn test to conﬁrm all tests pass.
2. Introduce a small mutation in the code (e.g., change a comparison or remove an excep-
tion check).
3. Re-run mvn test and observe which manual and LLM-based tests fail.
4. Discuss which mutants are killed only by LLM-generated tests and how this complements
the manual test suite.
This manual mutant experiment serves as a fallback and also builds intuition about mutation
testing when tools are not available.
